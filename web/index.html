One request. The agent runs the whole pipeline.

A data manager asks an agent to "get this CSV ready to submit." The agent composes the tools itself: inspect what is wrong, fix only what is safe to fix, then explain it in plain language.

FIRST

Inspect

validate_dataset()

Reads the CSV and returns every issue as structured JSON. Nothing is modified. This is the pre-check before anything is touched.

THEN

Fix what is safe

clean_dataset()

Applies information-preserving normalizations only. Writes cleaned.csv, a full changelog.json, and needs_review.json for the judgment calls.

FINALLY

Explain it

quality_report()

Returns a plain-language Markdown summary a non-technical data manager can read: counts by category and severity, with next steps.

Auto-fix, with a receipt

Every change is logged. Every judgment call is deferred.

The line TrialGate will not cross: a normalization that loses no information gets fixed and recorded. Anything that changes the meaning of a value is flagged for a human, never silently edited.

Three rules the tool will not break.

Trial data goes to regulators. An automated cleaner is only worth using if it can be trusted to touch exactly what it should, and nothing else.

01

Deterministic by design

The same input always produces the same output. There are no LLM calls anywhere in the rule logic, so a result can be reproduced, reviewed, and defended.

02

No silent edits

Every change lands in changelog.json with its row, column, before, after, and rule. A single missing entry is treated as a bug, not a rounding error.

03

When in doubt, do not fix

Auto-fixing is limited to normalization that loses no information. Anything that changes the meaning of a value goes to needs_review.json for a human.
